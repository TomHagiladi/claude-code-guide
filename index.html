🤖 בקצרה — מה זה בכלל?

Claude Code הוא כלי תוכנה שפותח על ידי חברת Anthropic (החברה שיצרה את Claude). בניגוד לצ'אטבוט רגיל שרק עונה על שאלות, Claude Code הוא סוכן (Agent) — כלומר, הוא לא רק מדבר, אלא גם עושה דברים:

  • 📂 קורא את כל הקבצים בפרויקט שלך — הוא מבין את המבנה, את הקוד, ואת ההקשר הכללי
  • ✏️ כותב ועורך קוד — יוצר קבצים חדשים, מתקן באגים, מוסיף פיצ'רים
  • 💻 מריץ פקודות בטרמינל — בדיקות, בניית פרויקט, ניהול Git ועוד
  • 🌐 מחפש באינטרנט — מוצא תיעוד, מידע עדכני ופתרונות
  • 🔗 מתחבר לכלים חיצוניים — Slack, GitHub, Google Drive ועוד דרך MCP

💡 במילים פשוטות: תארו לעצמכם מתכנת מנוסה שיושב לידכם ויכול לקרוא את כל הפרויקט שלכם, לכתוב קוד, להריץ בדיקות ולתקן בעיות — כל זה רק על ידי שאתם מסבירים לו מה אתם רוצים בשפה רגילה. זה Claude Code.

⚙️ איך זה עובד? — הלולאה האגנטית

כשנותנים ל-Claude Code משימה, הוא עובד בשלושה שלבים שחוזרים על עצמם עד שהמשימה מושלמת:

1

איסוף הקשר (Context Gathering)

Claude קורא קבצים, מחפש בפרויקט, ומבין מה קורה בקוד. הוא לא רואה רק את הקובץ הפתוח — הוא יכול לנווט בכל הפרויקט.

2

ביצוע פעולה (Taking Action)

Claude כותב קוד, עורך קבצים, מריץ פקודות, או מבצע כל פעולה שנדרשת.

3

בדיקה ואימות (Verification)

Claude מריץ בדיקות, בודק שהקוד עובד, ומתקן בעיות שהוא מוצא — עד שהתוצאה תקינה.

תהליך זה נקרא "הלולאה האגנטית" (Agentic Loop). Claude ממשיך לחזור על הצעדים האלה, לפעמים עשרות פעמים, עד שהמשימה מושלמת. ובכל שלב אפשר לעצור אותו ולתת לו כיוון אחר.

💪 יתרונות Claude Code

  • הבנה עמוקה של פרויקטים שלמים — Claude Code קורא ומבין את כל הקבצים בפרויקט, לא רק את הקובץ הפתוח. הוא מזהה קשרים בין רכיבים ויכול לעבוד על כמה קבצים במקביל.
  • עבודה אוטונומית — נותנים לו משימה והוא עובד לבד: מתכנן, כותב, בודק ומתקן. לא צריך להגיד לו צעד-צעד מה לעשות.
  • ללא נעילה לעורך מסוים — עובד בטרמינל, אז אפשר להשתמש בו עם כל עורך קוד (VS Code, Neovim, JetBrains ועוד).
  • איכות חשיבה גבוהה — המודל של Claude נחשב מהחזקים בעולם בהבנת קוד, דיבאגינג, ופתרון בעיות מורכבות.
  • זיכרון בין-שיחתי — דרך קבצי CLAUDE.md ו-Auto Memory, הוא זוכר העדפות, מוסכמות וסטנדרטים של הפרויקט.

  • שקיפות בעלויות — רואים בדיוק כמה טוקנים (tokens) נצרכו ומה העלות.
  • מתאים גם ללא-מתכנתים — אפשר לבנות אתרים, לנהל קבצים, לנתח נתונים ולאוטמט משימות — בלי לדעת לכתוב שורת קוד אחת.

  • הקשר מוגבל (Context Window) — בפרויקטים מאוד גדולים, Claude עלול לאבד מידע משיחות ארוכות. צריך לנהל את ההקשר בחכמה.
  • לא תמיד מדויק — כמו כל AI, הוא יכול לטעות, להמציא דברים (הזיות / Hallucinations), או לכתוב קוד שנראה נכון אבל מכיל באגים עדינים.

🛡️ סיכונים וסכנות — מה חשוב לדעת

שימוש בכלי AI לכתיבת קוד מגיע עם סיכונים שחשוב להכיר:

🔒 אבטחה

  • הרצת פקודות לא בטוחות — Claude Code יכול להריץ כל פקודה בטרמינל. במצב "אישור אוטומטי" הסיכון גדל. תמיד כדאי לבדוק מה הוא מתכוון להריץ.
  • פגיעויות אבטחה בקוד — AI יכול לכתוב קוד עם חולשות אבטחה (SQL Injection, XSS וכו'). מחקר של סטנפורד מצא שמפתחים שהשתמשו ב-AI כתבו קוד פחות מאובטח.
  • חשיפת מידע רגיש — הקוד שלכם נשלח לשרתים של Anthropic לעיבוד. לארגונים עם קוד רגיש, זה דורש שיקול.
  • מאגרי קוד זדוניים — בפברואר 2026 התגלו פגיעויות קריטיות: פתיחת פרויקט זדוני יכלה להוביל להרצת קוד זדוני ולגניבת מפתחות API.

🧠 תלות ולמידה

  • אובדן מיומנויות — הסתמכות יתר על AI עלולה למנוע פיתוח (או לגרום לאיבוד) של יכולת לדבג, לכתוב ולהבין קוד באופן עצמאי.
  • ביטחון מופרז — AI נותן הסברים בביטחון מלא, גם כשהוא טועה. זה יכול להוביל מתחילים לסמוך על קוד שהם לא באמת מבינים.

⚖️ משפטי וזכויות יוצרים

  • זכויות יוצרים — מודלי AI אומנו על קוד פתוח. יש תביעות משפטיות בנוגע לשימוש בקוד שעלול להפר רישיונות.

שלב 2.5 — אם הופיעה אזהרת PATH (חשוב מאוד ל-Windows!)

בסוף ההתקנה ב-Windows, אם רואים את ההודעה הצהובה הבאה — אל תדלגו עליה:

הודעה שמופיעה אחרי התקנה ב-PowerShell ‼ Setup notes: ● Native installation exists but C:\Users\YOUR_NAME\.local\bin is not in your PATH. Add it by opening: System Properties → Environment Variables → Edit User PATH → New → Add the path above. Then restart your terminal.

💡 מה זה PATH ולמה זה חשוב? PATH זו רשימת תיקיות ש-Windows סורק כשמקלידים פקודה (כמו claude). ההתקנה הצליחה ושמה את claude.exe בתיקייה C:\Users\YOUR_NAME\.local\bin, אבל Windows לא יודע לחפש שם. צריך "ללמד" אותו פעם אחת. בלי זה תקבלו "'claude' is not recognized" בכל פעם שתפעילו את Claude.

פתרון מהיר (מומלץ) — שורה אחת ב-PowerShell:

הדביקו את שתי השורות הבאות ב-PowerShell ולחצו Enter. זה מוסיף את התיקייה ל-User PATH לצמיתות:

PowerShell — תיקון קבוע $currentPath = [Environment]::GetEnvironmentVariable('PATH', 'User') [Environment]::SetEnvironmentVariable('PATH', "$currentPath;$env:USERPROFILE\.local\bin", 'User')

השורה הראשונה קוראת את ה-PATH הנוכחי. השנייה כותבת אותו חזרה עם תוספת של נתיב הבית שלכם + \.local\bin. אין צורך לערוך — Windows יחליף את $env:USERPROFILE אוטומטית.

אחרי הרצת הפקודה — חובה לבצע את שני השלבים הבאים:

  1. סגרו לחלוטין את חלון ה-PowerShell — לא טאב חדש, חלון חדש. השינוי נכנס לתוקף רק בחלונות חדשים.
  2. פתחו PowerShell חדש ובדקו עם הפקודה:
    claude --version

    אם רואים מספר גרסה (למשל 2.1.139) — הצליח! 🎉

⚙️ פתרון חלופי — דרך הממשק הגרפי (System Properties)

אם אתם מעדיפים לעבוד עם חלונות במקום פקודות, אפשר להוסיף את התיקייה ידנית:

  1. לחצו על מקש Windows במקלדת והקלידו environment variables (או בעברית: "משתני סביבה").
  2. בחרו Edit environment variables for your account — עריכת משתני סביבה עבור החשבון שלכם. חשוב: זו, ולא הגרסה של "System" — לזו אין צורך בהרשאות מנהל.
  3. בטבלה העליונה "User variables for [שם המשתמש]" — חפשו את השורה ששמה Path, לחצו עליה פעם אחת, ואז על כפתור Edit...
  4. בחלון החדש לחצו New מימין, ובשורה הריקה הדביקו את הנתיב המלא (להחליף YOUR_NAME בשם המשתמש שלכם):
    C:\Users\YOUR_NAME\.local\bin

    לא יודעים מה שם המשתמש? הוא רשום בהודעת ההתקנה עצמה — בדוגמה למעלה זה PC (כי הנתיב היה C:\Users\PC\.local\bin).

  5. לחצו OK בשלושת החלונות (מהפנימי לחיצוני).
  6. סגרו את חלון ה-PowerShell, פתחו חדש, ובדקו עם claude --version.

⏱️ פתרון זמני לסשן הנוכחי בלבד (במקרה שאתם ממהרים לעבוד עכשיו ותסדרו את ה-PATH הקבוע אחר כך):

$env:Path += ";$env:USERPROFILE\.local\bin"

הפקודה הזו מוסיפה את התיקייה ל-PATH רק לחלון ה-PowerShell הנוכחי. כשתסגרו אותו — תצטרכו להריץ אותה שוב. לכן מומלץ לעשות את התיקון הקבוע ברגע שיש לכם זמן.

⚠️ אם claude --version עדיין לא עובד

  • בדקו שהקובץ באמת קיים: הריצו Test-Path "$env:USERPROFILE\.local\bin\claude.exe" — אם חוזר True, הקובץ קיים והבעיה היא רק PATH.
  • ראו מאיפה Windows מנסה להריץ: where.exe claude — אם זה ריק, PATH עדיין לא נקלט (וצריך לסגור ולפתוח מחדש את החלון).
  • ודאו שהוספתם ל-User PATH, לא ל-System PATH — System PATH דורש הרשאות מנהל ומשפיע על כל המשתמשים. ברוב המקרים זה מיותר.

🍏 ב-macOS / Linux זה בדרך כלל לא בעיה — סקריפט ההתקנה מוסיף אוטומטית את ~/.local/bin ל-PATH דרך עריכת ~/.zshrc או ~/.bashrc. אם בכל זאת קיבלתם command not found: claude, הריצו:

Bash/Zsh (macOS & Linux) echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc source ~/.zshrc

ב-Linux החליפו ~/.zshrc ב-~/.bashrc.

📁 צעד 1 — איפה אני עובד? בחירת תיקיית פרויקט

▼

כלל הברזל: Claude Code עובד בתוך תיקייה אחת ספציפית. כל מה שהוא יקרא, ישנה או יבנה — יקרה בתוך התיקייה הזו. אסור לפתוח אותו מהשולחן עבודה או מתיקיית Downloads — בחרו תיקייה ייעודית.

אפשרות א' — פרויקט חדש מאפס (המומלץ למתחילים)

1

החליטו על שם. משהו תיאורי — "my-first-website", "todo-app", "klaf-bingo". אנגלית עדיפה, ללא רווחים, ללא תווים מיוחדים.

2

צרו תיקייה חדשה. Windows: File Explorer ← לכו ל-Documents ← לחיצה ימנית ← New ← Folder ← שם. Mac: Finder ← Documents ← לחיצה ימנית ← New Folder. הכי קל זה לפתוח את התיקייה ולראות אותה ריקה — זה הקנבס שלכם.

3

זה הכל בשלב זה. אל תיצרו עדיין שום קובץ בפנים — Claude יעשה את זה בשבילכם.

אפשרות ב' — פרויקט קיים על המחשב

אם כבר יש לכם תיקייה עם קוד — מצוין. דלגו לצעד הבא. Claude יקרא את מה שיש שם ויעבוד בהתאם.

אפשרות ג' — לעבוד על ריפוזיטורי מ-GitHub

אם המטרה היא לעבוד על פרויקט ב-GitHub (שלכם או של אחרים) — שתי דרכים:

  • דרך הדסקטופ / טרמינל / VS Code: צריך קודם להוריד אותו למחשב (פעולה שנקראת "clone"). Claude יעשה את זה בשבילכם — פשוט אמרו "תוריד לי את הריפו github.com/user/repo". הוא יבחר תיקייה, יוריד הכל ויפתח את הסשן בתוכה.
  • דרך הדפדפן (claude.ai/code): אין צורך להוריד — בוחרים את הריפו ישירות מתוך הממשק. Claude עובד עליו בענן.

💡 מה עדיף — Documents או Desktop? שני המקומות עובדים, אבל המלצה אישית: צרו תיקיית-על אחת בשם claude-projects בתוך Documents או בשורש הכונן (C:\) — וכל פרויקט חדש יהיה תת-תיקייה בתוכה. ככה הכל מסודר במקום אחד.

✍️ צעד 3 — הפרומפט הראשון שלכם

▼

זה הרגע. אתם רואים תיבת קלט. מה לכתוב?

💡 אין סגנון "נכון". Claude מבין עברית רגילה, אנגלית רגילה, ואפילו "תעשה לי משהו יפה". אבל ככל שתפרטו יותר, התוצאה תהיה קרובה יותר למה שדמיינתם.

אם זה פרויקט חדש — דוגמאות לפרומפט פתיחה

סוג פרויקט פרומפט פתיחה לדוגמה
אתר תדמית פשוט "תבנה לי דף נחיתה לסטודיו לעיצוב גרפי בשם 'אסטרא'. עברית RTL, צבעים חמים (חרדל וטרקוטה), שלוש סקציות: הסטודיו, שירותים, צור קשר. אחרי שתסיים, פתח את הקובץ בדפדפן ותראה לי."
אפליקציית משימות "תבנה לי אפליקציית רשימת משימות בעברית. אפשר להוסיף משימה, לסמן כבוצעה, ולמחוק. נשמר ב-localStorage כדי שלא ייעלם. עיצוב נקי, צבע ראשי כחול-טורקיז."
סקריפט אוטומציה "תכתוב לי סקריפט Python שעובר על כל קבצי ה-CSV בתיקייה הנוכחית, סופר כמה שורות יש בכל אחד, ויוצר דוח HTML יפה עם טבלה וסך-הכל. תריץ אותו אחרי שתכתוב כדי לוודא שזה עובד."
תיקון באג בקוד קיים "כשאני לוחץ על הכפתור 'שלח' בדף ההזמנה, כלום לא קורה. תבדוק את index.html ו-script.js, תמצא את הבעיה ותתקן. אל תשנה כלום אחר."

4 חוקי ברזל לפרומפט טוב

  1. תהיו ספציפיים. "תעשה את זה יפה" → "תוסיף הצללה, מרווחים גדולים יותר ופונט Heebo".
  2. תנו דוגמה. אם יש לכם אתר/תמונה/קוד שמוצא חן בעיניכם — צטטו את הקישור או גררו את התמונה לתוך השיחה.
  3. בקשו אימות. "אחרי שתסיים, תפתח את הדף בדפדפן ותראה לי שזה עובד" — שורה אחת שמשנה הכל. עוזרת ל-Claude לתפוס באגים לפני שתגלו אותם.
  4. סגרו את הגבולות. "תיגע רק ב-style.css" / "אל תיצור קבצים חדשים" — חוסכים שינויים לא רצויים.

⚠️ טעות מתחילים נפוצה: לבקש פרויקט ענק בפרומפט אחד ("תבנה לי רשת חברתית מלאה עם התחברות, צ'אטים, ושיתוף תמונות"). זה לעולם לא יעבוד טוב. במקום זה: תחילו בקטן. "תבנה לי דף התחברות" → רואים שזה עובד → "עכשיו תוסיף עמוד פרופיל" → וכן הלאה. כל צעד נבנה על הקודם.

✅ צעד 4 — איך לאשר ולדחות פעולות

▼

זה החלק שהכי מבלבל בהתחלה. Claude לא סתם כותב טקסט — הוא עושה דברים: יוצר קבצים, משנה קבצים, מריץ פקודות. לפני כל פעולה הוא ישאל אתכם.

מתי Claude שואל אישור

  • יצירת קובץ חדש — מציג לכם את התוכן ושואל אם לאשר
  • שינוי קובץ קיים — מציג השוואה לפני/אחרי (סימן + לשורה חדשה, סימן − לשורה שנמחקה)
  • הרצת פקודה בטרמינל — מציג את הפקודה ושואל "להריץ?"
  • גישה לאינטרנט / שירותים חיצוניים — תמיד שואל

שלוש האפשרויות שלכם

תשובה מה קורה מתי להשתמש
Yes (לאשר) הפעולה תתבצע, Claude ממשיך הלאה כשהשינוי נראה הגיוני
Yes, and don't ask again הפעולה תתבצע, ופעולות דומות יאושרו אוטומטית בהמשך הסשן כשבוטחים ב-Claude בנושא ספציפי (למשל "אפשר לערוך כל קובץ CSS בלי לשאול")
No (לדחות) הפעולה לא תתבצע. Claude יחכה שתסבירו מה לעשות אחרת כשהשינוי לא נראה הגיוני, או כשרוצים לכוון אותו אחרת

מצבי הרשאה — בחרו לפי רמת הביטחון שלכם

Claude Code מאפשר לכם להגדיר מראש "כמה לסמוך עליו". מחליפים בין המצבים בלחיצה על Shift+Tab. מסלול ברירת המחדל: Default → Accept Edits → Plan, וחוזר חלילה.

  • Default (ברירת מחדל) — שואל לפני כל פעולה משמעותית. המצב הכי בטוח, מומלץ למתחילים.
  • Accept Edits (acceptEdits) — מאשר אוטומטית עריכת קבצים ופקודות שמירת קבצים בסיסיות (mkdir, cp, mv), אבל עדיין שואל לפני הרצת פקודות אחרות. זה המצב היומיומי המומלץ אחרי שמתרגלים. תראו ⏵⏵ accept edits on בתחתית.
  • Plan mode — Claude רק חוקר ומתכנן, בלי לבצע שום שינוי. מצוין לפני שיוצאים למשימה גדולה. גם ב-Ctrl+G אפשר לפתוח את התוכנית בעורך טקסט ולערוך אותה לפני שClaud מתחיל.
  • Auto mode — קלאסיפייר חכם ברקע מאשר פעולות שגרתיות אוטומטית, וחוסם רק פעולות חשודות. לא זמין ב-Pro — דורש Max / Team / Enterprise / API, וגם דגם Sonnet 4.6 / Opus 4.6 / Opus 4.7. אם זה לא מופיע לכם — זו הסיבה.
  • Bypass permissions (YOLO) — הכל אוטומטי, בלי לשאול. לא מומלץ למתחילים — סיכון לנזק לקבצים. דורש דגל הפעלה מפורש (--permission-mode bypassPermissions) ולא יופיע ב-Shift+Tab בלי זה.

💡 איך לראות איפה אתם עכשיו? בתחתית המסך כתוב הסטטוס הנוכחי — "default mode", "auto mode" וכו'. אם אתם לא בטוחים — לחצו Esc וזה מחזיר אתכם למצב הבסיסי.

⏪ צעד 5 — לעצור, לחזור אחורה, ולנקות את השיחה

▼

הדבר הכי חשוב למתחילים: שום דבר לא בלתי הפיך. Claude יוצר אוטומטית נקודות שמירה (checkpoints) לפני כל פעולה, ואתם יכולים לחזור אחורה לכל רגע — גם אחרי שעות.

שלוש פעולות חירום — שעלו לזכור

המקרה הפתרון
Claude כותב כבר 30 שניות ואני רוצה לעצור אותו לחצו Esc — עוצר אותו מיד. השיחה נשמרת — אפשר להמשיך משם.
Claude עשה שינוי שלא רציתי לחצו Esc Esc (פעמיים) או הקלידו /rewind — ייפתח תפריט שמראה את כל הנקודות שאפשר לחזור אליהן. בוחרים נקודה, ו-Claude יחזיר את כל הקבצים למצב באותו רגע.
השיחה בלגן — Claude כבר לא מבין מה אני רוצה הקלידו /clear — מנקה את כל ההיסטוריה ומתחיל מאפס. הקבצים נשארים, רק השיחה מתחילה מחדש.

💡 למה /clear כל כך חשוב? Claude זוכר את כל השיחה — כל קובץ שקרא, כל שגיאה שראה, כל בקשה שלכם. ככל שהשיחה ארוכה יותר, ככה "הזיכרון" שלו מתמלא, ובשלב מסוים הוא מתחיל "לשכוח" דברים מההתחלה ולעשות שגיאות. כשמסיימים משימה ועוברים לחדשה — תמיד /clear. השוואה לחיים: כמו לפתוח טאב חדש בדפדפן במקום להמשיך לגלול לאחור 200 דפים.

הדרכים השונות לבטל

  • "תבטל את השינוי האחרון" — דרך מילולית, עובדת תמיד
  • Esc Esc או /rewind — תפריט ויזואלי של נקודות שמירה
  • Git revert — אם הפרויקט מחובר ל-Git, אפשר לבקש "תחזיר את כל הקבצים למה שהיו לפני שעה"

💾 צעד 6 — לסגור, לשמור, להמשיך מחר

▼

חלון השיחה עם Claude נסגר — מה קורה? הקבצים שלכם בטוחים. כל מה שClaud כתב לדיסק נשאר. השיחה עצמה נשמרת אוטומטית — אפשר לחזור אליה מחר.

איך לחזור לאותה שיחה

סביבה איך
טרמינל נווטו לתיקייה (cd ...) והריצו claude --continue כדי להמשיך את השיחה האחרונה, או claude --resume כדי לבחור מתוך רשימה של שיחות קודמות.
VS Code / דסקטופ פתחו את אותה תיקייה — תראו את היסטוריית השיחות בצד. לחיצה על שיחה ממשיכה משם.
דפדפן (claude.ai/code) תמיד תראו את כל הסשנים הפעילים בדף הראשי — לחיצה אחת כדי לחזור.

טיפ: תנו שם לסשן

💡 מה זה סקילים ופלאגינים? — הסבר פשוט

חשבו על Claude Code כמו טלפון חכם חדש. כשקונים אותו, הוא מגיע עם יכולות מובנות — אפשר להתקשר, לצלם, לגלוש באינטרנט. אבל הכוח האמיתי של הטלפון הוא באפליקציות שמתקינים עליו: וויז לניווט, אינסטגרם לתמונות, ווטסאפ להודעות.

באותו אופן בדיוק, Claude Code מגיע עם יכולות מרשימות — הוא יודע לכתוב קוד, לתקן באגים, לבנות אתרים. אבל אפשר להוסיף לו "אפליקציות" שנקראות סקילים ופלאגינים, שנותנות לו כוחות נוספים ומלמדות אותו לעבוד בדיוק כמו שאתם רוצים.

סקיל (Skill) — כמו כרטיס מתכון למטבח

דמיינו שיש לכם שף פרטי (זה Claude). הוא שף מעולה שיודע לבשל הכל. אבל אתם אוהבים שהפסטה תהיה תמיד אל-דנטה, שתמיד יהיה בזיל טרי, ושהרוטב יהיה מעגבניות טריות בלבד.

במקום לחזור ולהסביר את זה כל ערב — אתם כותבים כרטיס מתכון עם כל ההנחיות האלה, ותולים אותו במטבח. עכשיו, כל פעם שהשף מכין פסטה, הוא מסתכל על הכרטיס ועובד בדיוק לפי הטעם שלכם.

זה בדיוק מה שסקיל עושה. זה קובץ טקסט פשוט עם הוראות שמלמד את Claude איך לעשות משימה מסוימת. למשל: "כשמבקשים ממך לבנות אתר — תמיד תשתמש בעיצוב כהה, תוסיף אנימציות חלקות, ותבחר גופנים (פונטים) ישראליים".

מבחינה טכנית: סקילים נכתבים בקובץ טקסט בשם SKILL.md ונשמרים בתיקייה מיוחדת בשם .claude/skills/ בתוך תיקיית הפרויקט שלכם. אבל אל תדאגו — נסביר בהמשך בדיוק איך עושים את זה, צעד אחר צעד.

פלאגין (Plugin) — כמו ארגז כלים שלם

אם סקיל הוא כרטיס מתכון בודד, פלאגין הוא ספר בישול שלם — חבילה שמכילה בתוכה כמה מתכונים (סקילים) ואולי גם כלי מטבח מיוחדים.

למשל, הפלאגין superpowers מכיל בתוכו יותר מ-10 סקילים שונים: אחד שמלמד את Claude לתכנן לפני שהוא מתחיל לעבוד, אחד שמלמד אותו לחפש באגים בצורה שיטתית, ואחד שמלמד אותו לסקור קוד כמו מפתח מנוסה.

פלאגינים מופצים דרך שוק פלאגינים (Marketplace) — זה כמו חנות האפליקציות בטלפון. נכנסים, מחפשים מה שצריכים, ומתקינים בלחיצה אחת.

🚀 למה בכלל צריך סקילים ופלאגינים?

שאלה טובה! Claude Code כבר עובד מעולה בלי שום תוספות. אז למה להתעסק עם סקילים ופלאגינים? הנה חמש סיבות מעשיות:

  • 🎯 שהתוצאה תהיה תמיד כמו שאתם רוצים — בלי סקיל, כל פעם שתבקשו מ-Claude לבנות אתר, הוא יבחר עיצוב אחר, צבעים אחרים, גופנים אחרים. עם סקיל, אתם אומרים לו פעם אחת "ככה אני אוהב" — והוא יעבוד ככה בכל פעם. זה כמו ההבדל בין להזמין "קפה" בבית קפה חדש (מקבלים מה שבא להם) לעומת בית הקפה השכונתי שלכם שכבר יודע שאתם רוצים אמריקנו עם חלב שקדים.
  • 🔌 לחבר את Claude לשירותים שאתם משתמשים בהם — רוצים ש-Claude ישלח הודעה ב-Slack? יעלה קוד ל-GitHub? יעבוד עם העיצובים שלכם ב-Figma? בלי פלאגין הוא לא יודע לעשות את זה. עם הפלאגין המתאים — הוא מחובר ישירות ויכול לעבוד עם השירותים האלה.
  • ⏰ לחסוך זמן על הסברים חוזרים — במקום לכתוב כל פעם מחדש "תשתמש בעברית, תעצב מימין לשמאל, תשים את הלוגו שלי בפינה" — כותבים את זה פעם אחת בסקיל, והוא זוכר לבד בכל פעם.
  • 🧠 להפוך את Claude לחכם יותר בתחום שלכם — פלאגינים יכולים לתת ל-Claude הבנה עמוקה יותר של שפות תכנות מסוימות, או לחבר אותו לכלים שעוזרים לו למצוא בעיות בקוד. זה כמו לשלוח את השף שלכם לסדנת שוקולד — הוא כבר היה שף טוב, אבל עכשיו הוא מומחה גם בשוקולד.
  • 🤝 ליהנות ממה שאחרים כבר בנו — בשוק הפלאגינים יש עשרות כלים שמפתחים אחרים כבר יצרו ושיתפו בחינם. למה להמציא את הגלגל מחדש כשמישהו כבר עשה את העבודה?

🛒 איך מתקינים פלאגינים? — מדריך לשוק הפלאגינים

התקנת פלאגין היא קלה כמו התקנת אפליקציה בטלפון. יש שוק פלאגינים רשמי (Plugin Marketplace) — חנות מרכזית שבה אפשר לדפדף, לחפש ולהתקין פלאגינים מוכנים לשימוש. החנות מנוהלת על ידי Anthropic (החברה שיצרה את Claude), ומפתחים מכל העולם מעלים אליה פלאגינים.

🔍 דרך 1: גלישה בחנות (הדרך הקלה)

פתחו את Claude Code (בטרמינל או ב-VS Code) והקלידו את הפקודה הבאה. שימו לב לסלש (/) בהתחלה — זה אומר ל-Claude שזו פקודה מיוחדת ולא שאלה רגילה:

הקלידו בתוך Claude Code /plugin

זה יפתח תפריט עם שלוש לשוניות (כמו טאבים בדפדפן):

  • Installed (מותקנים) — רשימת כל הפלאגינים שכבר התקנתם. כאן תוכלו גם להסיר פלאגינים שאתם לא צריכים יותר.
  • Discover (גלו חדשים) — זו ה"חנות" עצמה. כאן תראו עשרות פלאגינים מסודרים לפי קטגוריות. תוכלו לגלול, לקרוא תיאורים, ולהתקין מה שמעניין אתכם.
  • Add (הוספה ידנית) — אם מישהו נתן לכם כתובת של פלאגין מסוים (למשל מ-GitHub), תוכלו להדביק אותה כאן.

⚡ דרך 2: התקנה ישירה (למי שיודע מה הוא רוצה)

אם כבר יודעים את שם הפלאגין (למשל, קראתם עליו כאן), אפשר להתקין ישירות בלי לגלוש בחנות. מקלידים את פקודת הסלש הבאה ב-Claude Code:

הקלידו בתוך Claude Code /plugin install frontend-design@claude-plugins-official

בואו נפרק את הפקודה הזו כדי שתבינו מה קורה:

  • /plugin install — אומר ל-Claude: "תתקין פלאגין"
  • frontend-design — שם הפלאגין שרוצים להתקין
  • @claude-plugins-official — שם החנות. החנות הרשמית של Anthropic נקראת "claude-plugins-official"

אחרי ההתקנה, הפלאגין מוכן לשימוש מיד — אין צורך לעשות שום דבר נוסף!

📦 איפה הפלאגין יהיה זמין? — שלושה מצבים

כשמתקינים פלאגין, אפשר לבחור באיזה "טווח" הוא יהיה זמין. רוב המשתמשים לא צריכים לדאוג מזה — ברירת המחדל (User) מתאימה כמעט תמיד. אבל הנה שלושת המצבים למקרה שתרצו לדעת:

מצב מה זה אומר בשפה פשוטה מתי להשתמש
User
(ברירת מחדל)		 הפלאגין עובד בכל מקום, בכל פרויקט — כמו אפליקציה שמותקנת על הטלפון כמעט תמיד. זה מה שתרצו
Project הפלאגין שמור בתוך הפרויקט — אם תשתפו את הפרויקט עם מישהו, גם הוא יקבל את הפלאגין כשעובדים בצוות על פרויקט משותף
Local הפלאגין עובד רק אצלכם, רק בפרויקט הספציפי הזה כשרוצים לנסות פלאגין בלי שזה ישפיע על שום דבר אחר

🔄 עדכון רשימת הפלאגינים

כמו חנות אפליקציות, גם שוק הפלאגינים מתעדכן כל הזמן — מוסיפים פלאגינים חדשים ומעדכנים קיימים. כדי לוודא שאתם רואים את הרשימה העדכנית ביותר:

הקלידו בתוך Claude Code /plugin marketplace update claude-plugins-official

מומלץ להריץ את הפקודה הזו פעם בשבוע-שבועיים, כדי לוודא שאתם לא מפספסים פלאגינים חדשים ומעניינים.

⭐ פלאגינים מומלצים — מאיפה כדאי להתחיל?

בשוק הפלאגינים יש עשרות אפשרויות, וזה יכול להרגיש מבלבל. הנה רשימה של הפלאגינים הכי שימושיים, מחולקים לפי מה שהם עושים. לא צריך להתקין את כולם — בחרו רק את מה שרלוונטי עבורכם.

🧠 שדרוג הדרך שבה Claude עובד

  • superpowers — מומלץ במיוחד! חבילת סקילים שמשנה את הדרך שבה Claude עובד. במקום לקפוץ ישר לכתיבת קוד, הוא קודם מתכנן, בודק, ורק אז מבצע. זה כמו ההבדל בין שיפוצניק שמתחיל לפרק קירות בלי תכנית, לעומת אדריכל שקודם מצייר תכניות ורק אז מתחיל לבנות.
  • explanatory-output-style — גורם ל-Claude להסביר כל דבר שהוא עושה: מה הוא שינה, למה, ומה זה אומר. מושלם למי שלומד ורוצה להבין מה קורה, לא רק לקבל תוצאה.

🔍 מה יש בתוך פלאגין? — חמשת הרכיבים

פלאגין הוא כמו ארגז כלים, ובתוכו יכולים להיות כמה סוגים של "כלים". רוב הפלאגינים מכילים רק סקילים (הרכיב הראשון ברשימה), אבל חלקם מכילים גם רכיבים נוספים. הנה הסבר על כל סוג:

רכיב מה זה בשפה פשוטה דוגמה מהחיים
סקילים
(Skills)		 "כרטיסי מתכון" — קבצי טקסט עם הוראות שמלמדים את Claude איך לעשות דברים. זה הרכיב הכי נפוץ והכי פשוט. "כשבונים אתר — תמיד תשתמש בגופנים מיוחדים, תוסיף אנימציות, ותעצב בסגנון כהה"
סוכנים
(Agents)		 "עוזרים מתמחים" — גרסאות מותאמות של Claude שמקבלות כלים ספציפיים לתחום מסוים. כמו לשכור עובד שמתמחה בתחום אחד. סוכן שמתמחה ב-Slack: הוא יודע לקרוא הודעות, לשלוח תגובות, ולחפש בשיחות ישנות

קובץ ההגדרות — "תעודת הזהות" של הפלאגין

כל פלאגין חייב קובץ בשם plugin.json בתוך תיקיית .claude-plugin/ שמסביר ל-Claude Code מה הפלאגין. זה כמו תווית על קופסה. הנה דוגמה מינימלית:

.claude-plugin/plugin.json — תעודת הזהות של הפלאגין { "name": "my-awesome-plugin", "description": "A plugin that helps build Hebrew websites", "version": "1.0.0", "author": { "name": "Your Name" } }

מה כל שורה אומרת:

  • name — שם הפלאגין (באנגלית, בלי רווחים). הוא גם משמש כקידומת לכל סקיל בתוך הפלאגין — למשל /my-awesome-plugin:hebrew-website
  • description — תיאור קצר של מה הפלאגין עושה
  • version — מספר גרסה (מתחילים מ-1.0.0). אם משמיטים אותו ומפיצים דרך git — ה-SHA של הקומיט משמש כגרסה
  • author — מי יצר את הפלאגין (אופציונלי)

💡 שינוי חשוב מ-2025: בעבר היה צריך לרשום במניפסט את כל הסקילים והסוכנים בשדה components. היום זה לא נחוץ — Claude Code מזהה אוטומטית כל מה שנמצא בתוך התיקיות skills/, agents/, hooks/, .mcp.json ו-.lsp.json. אתם רק שמים את הקבצים במקום הנכון, וזה עובד.

בדיקה לוקאלית של פלאגין לפני הפצה

תוך כדי פיתוח, אפשר להריץ את Claude Code עם דגל --plugin-dir כדי לטעון את הפלאגין מתיקייה מקומית בלי להעלות אותו לחנות:

💡 חוקי האגודל הם נקודת התחלה — לא חוקים מקודשים. לפעמים שווה לסטות מהם בכוונה. המטרה היא לפתח אינטואיציה, לא להיצמד למתכון. ככל שתעבדו יותר, תזהו מתי איזה כלל חשוב.

✅ 1. תנו ל-Claude דרך לאמת את עצמו — הכלל הכי חשוב

הדבר היחיד בעל ההשפעה הגדולה ביותר על איכות התוצאה. Claude עובד הרבה יותר טוב כשהוא יכול להריץ בדיקות בעצמו, להשוות צילומי מסך, ולוודא שהתוצאה אכן נכונה.

בלי קריטריון להצלחה, Claude עלול לייצר משהו שנראה נכון אבל בפועל לא עובד — ואתם הופכים להיות הלולאה היחידה שבודקת. כל באג דורש את תשומת הלב שלכם.

השיטה פרומפט חלש פרומפט חזק
תנו קריטריוני בדיקה "תכתוב פונקציה שמאמתת כתובות מייל" "תכתוב פונקציה validateEmail. מקרי בדיקה: user@example.com → true, invalid → false, user@.com → false. תריץ את הבדיקות אחרי שתכתוב."

💡 צורות אימות אחרות: מערכת בדיקות (tests), כלי לבדיקת תקינות קוד (linter), פקודה בטרמינל שמדפיסה תוצאה. ככל שלכלי האימות שלכם יש פחות "רעש" — Claude מתפקד טוב יותר. שיפור בכלי האימות הוא תמיד השקעה משתלמת.

🧭 2. חקרו תחילה, תכננו, ואז קודו (Plan Mode)

להניח ל-Claude לקפוץ ישר לקוד יכול לייצר פתרון לבעיה הלא נכונה. השתמשו ב-Plan Mode כדי להפריד חקירה מביצוע.

איך נכנסים ל-Plan Mode? לחיצה כפולה על Shift+Tab. תראו את הסטטוס "plan mode" בתחתית — Claude לא יבצע שום שינוי, רק יחקור, יקרא קבצים, ויבנה תוכנית.

תהליך עבודה מומלץ — 4 שלבים

1

חקירה (Plan Mode פעיל):

תקרא את src/auth ותסביר לי איך אנחנו מנהלים sessions והתחברות. גם תבדוק איך אנחנו מנהלים משתני סביבה לסודות.
2

תכנון (Plan Mode פעיל):

אני רוצה להוסיף התחברות עם Google OAuth. אילו קבצים צריכים להשתנות? איך זרימת ה-session תיראה? תיצור תוכנית מפורטת.
3

יישום (יציאה מ-Plan Mode — לחיצה נוספת על Shift+Tab):

תיישם את התוכנית. תכתוב tests ל-callback handler, תריץ אותם, ותתקן כל מה שנכשל.
4

קומיט:

תעשה commit עם הודעה תיאורית ותפתח PR

💡 מתי לדלג על Plan Mode? לתיקון קטן וברור (typo, שינוי שם משתנה, תוספת לוג) — לכו ישר לביצוע. Plan מתאים כשאתם לא בטוחים בגישה, כששינוי נוגע במספר קבצים, או כשאתם לא מכירים את הקוד. כלל אצבע: אם תוכלו לתאר את ה-diff במשפט אחד — דלגו על התכנון.

📍 3. תנו הקשר ספציפי בפרומפט

ככל שההוראות מדויקות יותר — כך תצטרכו לתקן פחות. Claude מסיק כוונות, אבל הוא לא קורא מחשבות. הצביעו על קבצים ספציפיים, ציינו אילוצים, והפנו לדפוסים קיימים.

שיטה לפני אחרי
תחמו את המשימה "תוסיף tests ל-foo.py" "תכתוב test ל-foo.py למקרה הקצה שבו המשתמש לא מחובר. אל תשתמש ב-mocks."
הצביעו על מקור "למה ל-ExecutionFactory יש API מוזר?" "תעבור על ההיסטוריה של ExecutionFactory ב-git ותסביר איך ה-API שלו התפתח."
הפנו לדפוסים קיימים "תוסיף widget לוח שנה" "תסתכל איך widgets קיימים מיושמים בדף הבית. HotDogWidget.php הוא דוגמה טובה. תעקוב אחרי אותו דפוס ותיישם widget לוח שנה. בלי ספריות חיצוניות חדשות."
תארו את התסמין "תתקן את הבאג בהתחברות" "משתמשים מדווחים שהתחברות נכשלת אחרי session timeout. תבדוק את זרימת ה-auth ב-src/auth/, במיוחד את ה-token refresh. תכתוב test שמשחזר את הבעיה, אז תתקן."

🎁 דרכים להעשיר את ההקשר

  • הצביעו על קבצים עם @ — במקום לתאר איפה הקוד, כתבו @src/auth.ts — Claude יקרא את הקובץ לפני שיענה
  • הדביקו תמונות ישירות — Ctrl+V או גרירה לתוך השיחה
  • תנו URLs — לתיעוד, ל-API references, לאתר עם דוגמה. השתמשו ב-/permissions כדי לאפשר דומיינים שכיחים מראש
  • "צינור" נתונים — בטרמינל: cat error.log | claude שולח את תוכן הקובץ ישירות
  • תנו ל-Claude לחפש לבד — "תמצא בעצמך את הקובץ הרלוונטי" עובד טוב כשאתם לא בטוחים איפה הקוד

🪟 4. נהלו את חלון ההקשר באגרסיביות

רוב המלכודות בעבודה עם Claude Code נובעות ממגבלה אחת: חלון ההקשר מתמלא מהר, והביצועים יורדים ככל שהוא מתמלא.

"חלון ההקשר" של Claude מחזיק את כל השיחה — כל הודעה, כל קובץ שקרא, כל פלט פקודה. סשן debug ארוך אחד יכול לצרוך עשרות אלפי טוקנים. ככל שהוא יותר מלא, ככה Claude יותר "שוכח" הוראות מההתחלה ועושה יותר טעויות.

אסטרטגיות יעילות

  • /clear בין משימות לא קשורות — הכלל הכי חשוב. סיימתם משימה? לפני שעוברים למשימה הבאה — /clear. תמיד.
  • /compact כשהשיחה ארוכה מדי — Claude מסכם את עצמו, שומר את החשוב ומשחרר זיכרון.
  • /compact <הוראה> לדחיסה ממוקדת — למשל "/compact תשמור רק את השינויים ב-API".
  • /btw לשאלות צדדיות — הקלידו /btw מה זה useEffect? והתשובה תופיע בחלון חולף, בלי להיכנס להיסטוריית השיחה. מצוין להבהרות מהירות.
  • תת-סוכנים (Subagents) לחקירות — במקום שClaud יקרא 30 קבצים בעצמו ויצרוך את כל ההקשר, תגידו "תשתמש בתת-סוכן כדי לחקור איך המערכת מטפלת ב-X". הוא יחקור בחלון נפרד ויחזור עם סיכום קצר.

💡 סימני אזהרה שההקשר מתחיל להיות בעייתי: Claude עונה בשגיאה על שאלה שכבר ענה עליה נכון; חוזר על אותו תיקון פעמיים; "שוכח" משתנים שהגדרתם בתחילת השיחה; הפלט מתחיל להיות לא קשור לבקשה. כשרואים שניים מאלה — /clear מיידי.

🎓 פיתוח האינטואיציה שלכם

הדפוסים במדריך הזה הם נקודת התחלה — לא חוקים מקודשים. הם עובדים טוב בכלל, אבל לא בהכרח אופטימליים לכל מצב.

לפעמים כדאי לתת להקשר להצטבר כי אתם עמוק בבעיה מסובכת אחת וההיסטוריה חשובה. לפעמים כדאי לדלג על תכנון ולתת ל-Claude לחקור — כי המשימה עצמה אקספלורטיבית. לפעמים פרומפט מעורפל הוא בדיוק הנכון — כי אתם רוצים לראות איך Claude יפרש את הבעיה לפני שתצמצמו.
